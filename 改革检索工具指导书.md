# 改革检索工具指导书

## 目标
把当前智慧农业文献检索 Web 工具，逐步改造成更接近 OpenAlex 的检索体验：
- 先让用户“更容易搜到、看得懂、筛得快”
- 再让页面“更像数据库产品，而不是普通列表页”
- 最后再扩展到统计、分析、推荐等能力

## 总原则
1. 只做能独立完成并验证的小步骤，不做一次性大改造。
2. 每一步都必须有明确输入、明确输出、明确验收标准。
3. 每一步只改一个核心点，避免同时改布局、逻辑、样式和交互。
4. 先保证稳定，再追求美观，再追求分析感。
5. 所有改动都必须可回退、可验证、可继续拆分。

## OpenAlex 对标方向
要接近 OpenAlex，重点不是“功能堆很多”，而是这 4 个特征：
- 搜索入口足够清晰
- 结果展示足够紧凑
- 筛选条件足够直观
- 数据层级足够清楚

## 适合长期执行的小步骤

### 一、页面骨架改造

#### 步骤 1：先统一页面标题区
目标：让首页第一眼像数据库产品。

可成功实现的动作：
- 保留纯文字标题
- 去掉装饰性图标
- 统一标题、说明、副标题的字号和间距
- 控制标题区高度不要太大

验收标准：
- 页面顶部不杂乱
- 标题区不占用过多垂直空间
- 搜索框能尽快进入用户视野

#### 步骤 2：把搜索框固定为主视觉
目标：让用户一进页面就知道先搜什么。

可成功实现的动作：
- 搜索框放在页面上方中心位置
- 只保留一个主搜索框
- 搜索按钮大小统一
- 输入提示简短明确

验收标准：
- 用户不用找功能入口
- 页面最显眼的是搜索，不是说明文字

#### 步骤 3：把结果区和筛选区分开
目标：结果和控制区职责清楚。

可成功实现的动作：
- 左边放结果列表
- 右边放筛选和统计信息
- 不在结果卡片里堆放太多控制项

验收标准：
- 用户能一眼分辨“看结果”和“改条件”
- 页面不会因为信息混杂而显乱

### 二、结果卡片改造

#### 步骤 4：压缩单条结果的视觉高度
目标：让结果更像 OpenAlex 那种“紧凑数据库条目”。

可成功实现的动作：
- 标题只占一行或两行
- 作者、年份、来源、引用数合并成一行或两行
- 摘要默认折叠
- DOI 放到次级位置

验收标准：
- 一屏能看到更多结果
- 每条结果信息层次清楚

#### 步骤 5：统一字段显示顺序
目标：让用户形成稳定阅读习惯。

推荐顺序：
- 标题
- 作者
- 年份
- 来源
- 引用数
- DOI
- 摘要

验收标准：
- 所有结果卡片格式一致
- 用户浏览不需要重新学习布局

#### 步骤 6：把操作按钮放到固定区域
目标：避免按钮分散在各处。

可成功实现的动作：
- 翻译按钮固定在卡片右侧或底部
- DOI跳转保持统一位置
- 展开摘要放在同一层级

验收标准：
- 用户不会到处找按钮
- 卡片操作逻辑一致

### 三、筛选系统改造

#### 步骤 7：先保留最核心筛选
目标：不让筛选变成复杂面板。

可成功实现的动作：
- 先只保留年份、引用数、来源这类基础筛选
- 把高级筛选放入折叠区
- 默认显示最常用筛选项

验收标准：
- 新用户不会被筛选项吓到
- 老用户能快速调整条件

#### 步骤 8：把筛选条件变成可见状态
目标：让用户知道当前结果是怎么筛出来的。

可成功实现的动作：
- 在结果区上方显示当前筛选条件标签
- 显示当前排序方式
- 显示过滤后结果数量

验收标准：
- 用户知道自己正在看什么结果集
- 不会误以为结果是“系统随机给的”

#### 步骤 9：把排序方式做成可切换项
目标：让结果排序像 OpenAlex 一样可控。

可成功实现的动作：
- 相关度排序
- 引用数排序
- 年份排序
- 默认给出一个最合理排序

验收标准：
- 用户能主动控制列表顺序
- 不需要改代码就能切换排序

### 四、检索质量改造

#### 步骤 10：先做“硬过滤”再做“软排序”
目标：先保证相关，再谈好看。

可成功实现的动作：
- 只保留标题或摘要命中关键词的文献
- 排除只在作者、来源、参考文献中命中的结果
- 再对保留结果做综合排序

验收标准：
- 结果更干净
- 低相关文献明显减少

#### 步骤 11：把同义词扩展限制在小范围
目标：扩展召回，但不发散。

可成功实现的动作：
- 只扩展农业、育种、基因组学等核心词
- 每次只加少量同义词
- 扩展词必须能解释为什么相关

验收标准：
- 能提高召回
- 不会把结果带偏

#### 步骤 12：把相关度分数显示为内部字段
目标：方便调参，但不干扰用户。

可成功实现的动作：
- 先在代码里算出相关度
- 可选展示给调试模式
- 普通用户默认不显示复杂分数

验收标准：
- 后续调参有依据
- 页面不会变成“技术面板”

### 五、OpenAlex风格增强

#### 步骤 13：增加“结果统计感”
目标：让页面更像数据库检索平台。

可成功实现的动作：
- 显示总结果数
- 显示当前页结果数
- 显示检索耗时
- 显示来源分布

验收标准：
- 用户能感受到系统“在处理数据”
- 页面有数据库产品感

#### 步骤 14：增加结果导航感
目标：让用户在大量结果中更容易浏览。

可成功实现的动作：
- 分页
- 当前页高亮
- 上一页/下一页按钮清晰
- 每页固定条数

验收标准：
- 结果多时不乱
- 翻页体验稳定

#### 步骤 15：增加轻量标签信息
目标：快速帮助用户判断结果是否值得看。

可成功实现的动作：
- 来源 API 标签
- 年份标签
- 引用数标签
- 语言标签或翻译标签

验收标准：
- 用户扫一眼就知道文献的大概价值
- 不需要点开每条结果

### 六、交互体验改造

#### 步骤 16：让翻译功能只出现在合适的地方
目标：减少干扰，提高可用性。

可成功实现的动作：
- 只有英文文献显示翻译按钮
- 中文文献不显示
- 翻译状态可切换回原文

验收标准：
- 按钮不会乱出现
- 用户不会误操作

#### 步骤 17：让空结果页给出可执行建议
目标：不要只说“没有结果”。

可成功实现的动作：
- 提示换关键词
- 提示放宽阈值
- 提示尝试同义词

验收标准：
- 用户知道下一步该怎么做
- 空状态也有引导价值

#### 步骤 18：让加载状态更像产品
目标：避免“卡死感”。

可成功实现的动作：
- 显示加载中
- 显示正在合并结果
- 显示正在筛选/排序

验收标准：
- 用户知道系统在工作
- 网络慢时仍有反馈

### 七、推荐执行顺序

#### 第一阶段：先做结构
1. 改页面顶部
2. 改搜索区
3. 改结果区分栏
4. 改单条卡片结构

#### 第二阶段：再做控制
5. 改筛选区
6. 改排序区
7. 改结果状态提示

#### 第三阶段：再做增强
8. 改翻译交互
9. 改统计面板
10. 改分页和导航

#### 第四阶段：最后做精修
11. 调整颜色
12. 调整间距
13. 调整字体权重
14. 调整响应式布局

## 每次开发前的检查清单
- 这一步是不是只改一个核心点？
- 改完后能不能立刻看到效果？
- 能不能用一句话说明完成标准？
- 会不会影响现有检索功能？
- 能不能失败后容易回退？

## 每次开发后的验收清单
- 页面是否仍能正常打开
- 搜索是否仍能正常执行
- 结果是否更清晰
- 低相关内容是否减少
- 页面是否更接近 OpenAlex 的数据库感

## 禁止事项
- 不要一次性重做所有页面
- 不要同时改太多交互逻辑
- 不要为了像 OpenAlex 而牺牲稳定性
- 不要加入会让用户看不懂的复杂操作
- 不要把页面做得像海报，不像工具

## 备注
这份指导书的用途不是立刻执行所有内容，而是把后续优化拆成一连串“能做成、能验收、能继续推进”的小步骤。以后每次迭代，都先从这里挑 1 到 2 步做。

## 上线与限制解决方案

### 一、先上线的最小可行路线
目标：先让项目稳定跑到公网，再逐步优化体验。

可成功实现的动作：
- 使用 GitHub 仓库作为唯一代码源
- 采用 Streamlit Cloud 进行首版部署
- 入口页面只保留 `main.py`
- 依赖只保留必要库，避免引入重型包
- 把 API 密钥放到 Streamlit Secrets，而不是写死在代码里

验收标准：
- 代码提交后可以自动触发部署
- 用户能通过公网地址访问页面
- 页面打开后能正常搜索、展示、筛选

### 二、解决 Streamlit Cloud 冷启动慢
问题：免费托管会休眠，首次打开会慢。

可成功实现的方法：
- 把重计算延后到点击搜索后再执行
- 减少启动时的预加载内容
- 首页尽量只渲染轻量文本和输入控件
- 让页面首次加载不依赖大文件、模型或本地缓存

验收标准：
- 首页能快速打开
- 没有搜索时页面尽量轻量
- 首次访问慢，但后续交互不再明显卡顿

### 三、解决外部 API 不稳定
问题：PubMed、Crossref、OpenAlex 都可能超时或限流。

可成功实现的方法：
- 每个 API 都设置超时
- 三个 API 并发请求，减少总等待时间
- 对失败请求做降级处理，不让整个页面崩溃
- 对结果先合并再去重，减少重复内容
- 对关键查询结果做缓存，避免重复请求同一词

验收标准：
- 单个 API 报错时页面仍能返回其他来源结果
- 网络波动时页面不会崩溃
- 重复搜索同一关键词时响应更稳

### 四、解决结果太少或太杂
问题：严格过滤后可能结果少，扩展后可能又发散。

可成功实现的方法：
- 先做标题/摘要硬过滤
- 再做同义词小范围扩展
- 扩展检索只保留少量阶段，不无限发散
- 用相关度分数排序，优先保留高匹配结果
- 通过“自动放宽阈值”只补足必要数量

验收标准：
- 低相关结果明显减少
- 单关键词尽量稳定达到 10 篇
- 多关键词不被过度扩展污染

### 五、解决翻译功能的部署限制
问题：在线翻译依赖第三方服务，可能失败或变慢。

可成功实现的方法：
- 翻译失败时直接回退原文，不阻断页面
- 只对英文文献显示翻译按钮，减少无效请求
- 翻译结果做会话级缓存，避免重复翻译同一文献
- 只翻译标题、摘要、来源等核心字段，不翻译全部元数据

验收标准：
- 翻译失败不会影响检索和展示
- 已翻译内容再次打开时更快
- 中文文献不出现翻译按钮

### 六、解决页面越来越复杂
问题：随着功能增加，页面容易变乱。

可成功实现的方法：
- 把筛选面板固定在侧边栏
- 把结果卡片保持统一模板
- 把统计信息放在结果区顶部
- 把调试信息隐藏在开发模式中
- 每加一个新功能都先确定显示位置

验收标准：
- 页面结构始终清楚
- 新功能加入后不会挤乱主流程
- 用户能稳定找到搜索、筛选、查看结果这三件事

### 七、上线前检查清单
- `main.py` 是否是唯一入口
- `requirements.txt` 是否只保留必要依赖
- 是否已经处理 API 超时和异常
- 是否没有写死敏感信息
- 首页加载是否足够轻量
- 搜索结果是否有去重和排序
- 翻译失败是否会自动回退原文

### 八、后续优化顺序
1. 先保证公网能访问
2. 再保证结果稳定
3. 再提升页面像 OpenAlex
4. 再做缓存和速度优化
5. 最后再考虑更复杂的 AI 或图谱功能

### 九、当前版本上线验收清单（可直接执行）
目标：把当前“已按 OpenAlex 风格改造”的版本快速且稳定上线。

执行步骤：
1. 确认分支代码可运行：本地执行 `python -m compileall main.py`，无报错。
2. 确认依赖完整：`requirements.txt` 包含 `streamlit`、`requests`、`biopython`、`deep-translator`。
3. 确认入口文件：部署入口设置为 `main.py`。
4. 确认配置文件：`.streamlit/config.toml` 存在且未写入敏感信息。
5. 确认异常处理：三大 API 的超时和异常均不会导致页面崩溃。
6. 确认弱网提示：任一数据源异常时，页面会提示“自动降级到可用结果”。
7. 确认核心交互：翻页、页码跳转、排序切换、重置筛选、回到第1页均可用。
8. 确认翻译交互：仅英文文献显示翻译按钮，中文文献不显示。
9. 确认空状态：无结果时能看到明确可执行建议。
10. 确认展示密度：结果卡片为紧凑布局，摘要可展开/收起。

上线后 10 分钟巡检：
1. 用 3 个关键词实测（1 个单关键词、2 个多关键词）。
2. 观察检索耗时与结果数是否稳定。
3. 检查侧边栏统计（来源分布、年份分布）是否正常刷新。
4. 检查 DOI 链接是否可点击跳转。
5. 检查移动端是否出现按钮挤压或样式错位。

出现问题时的最小回退策略：
1. 先回退到上一个可运行提交，不做大面积热修。
2. 优先保证搜索、筛选、结果展示三条主链路可用。
3. 翻译和统计可暂时降级，不阻断主流程。
